Quant Studio SageMaker Setup
============================

Recommended path: VS Code Remote to SageMaker
---------------------------------------------

Use this file when you want to run Quant Studio on SageMaker compute while
using your local VS Code desktop IDE.

The recommended workflow is:

1. Open the SageMaker space through local VS Code Remote access.
2. Open a VS Code terminal connected to the SageMaker machine.
3. Clone or pull the repo on the SageMaker filesystem.
4. Bootstrap the project once:

   cd /home/sagemaker-user/quant_workflow
   bash scripts/bootstrap_sagemaker.sh

5. Start Streamlit from the same remote VS Code terminal:

   bash scripts/run_sagemaker_streamlit.sh

6. In local VS Code, open the Ports / Forwarded Ports panel.
7. Forward remote port 8501.
8. Open the forwarded local URL, usually:

   http://localhost:8501

Why localhost works in this setup
---------------------------------

Streamlit is running on the SageMaker machine, not on your laptop. VS Code
Remote forwards SageMaker port 8501 back to your local machine. After the port
is forwarded, your browser's localhost:8501 points through VS Code to the
remote SageMaker Streamlit process.

Do not use BASE_URL_PATH for VS Code Remote
-------------------------------------------

When using local VS Code Remote plus port forwarding, start the app with:

   bash scripts/run_sagemaker_streamlit.sh

Do not use:

   BASE_URL_PATH=/jupyterlab/default/proxy/absolute/8501 bash scripts/run_sagemaker_streamlit.sh

The BASE_URL_PATH form is only for SageMaker JupyterLab browser proxy URLs.

What the scripts do
-------------------

- scripts/bootstrap_sagemaker.sh
  - checks for Python 3.11+
  - creates a repo-local .sagemaker_venv virtual environment
  - installs the dependencies from requirements-sagemaker.txt into that venv
  - installs Quant Studio from the local source tree
  - checks Plotly static chart export support for optional per-figure PNG
    exports and attempts to install Chrome for Kaleido unless disabled with
    INSTALL_PLOTLY_CHROME=0

- scripts/run_sagemaker_streamlit.sh
  - uses .sagemaker_venv automatically when it exists
  - runs Streamlit on 0.0.0.0:8501
  - accepts BASE_URL_PATH only for browser proxy fallback scenarios
  - keeps the same upload-size settings used by the Windows launcher

Why the virtual environment matters
-----------------------------------

SageMaker JupyterLab and Code Editor images include their own Jupyter and
Notebook packages. Installing application dependencies directly into that base
environment can surface resolver warnings such as:

   notebook 7.4.4 requires jupyterlab<4.5,>=4.4.4, but you have jupyterlab 4.2.5

Quant Studio does not need to alter SageMaker's IDE packages. The bootstrap
script therefore installs the app into .sagemaker_venv so the Streamlit app
runtime is isolated from SageMaker's Jupyter runtime.

Loading data in SageMaker
-------------------------

For remote runs, place CSV, Excel, or Parquet files in the repo-level
Data_Load/ directory, then choose Select from Data_Load in Step 1, Dataset &
Schema, under the data-source controls. This avoids browser-upload friction
and records the selected file name, suffix, size, and modified time in
reproducibility_manifest.json.

Data_Load/ is ignored by git except for .gitkeep, so large or private datasets
are not pushed to GitHub.

For multi-GB files, prefer Parquet over CSV or Excel. If you start with CSV,
use the Data_Load conversion button or enable Large data mode plus Convert CSV
file-path inputs to Parquet before ingestion.

Large dataset sizing guidance
-----------------------------

The 50 GB Streamlit upload setting is a configured ceiling, not a memory
guarantee. Pandas can require several times the source-file size after parsing,
and diagnostics can create additional copies.

Conservative starting points:

- Under 1 GB: use at least 16 GB RAM.
- 1-5 GB CSV: use 32-64 GB RAM and Large data mode.
- 5-10 GB CSV: use 96-128 GB RAM, prefer Data_Load over browser upload,
  convert to Parquet first, and use sampled exports.
- Parquet usually reduces disk and parse cost, but the decompressed dataframe
  can still be large in memory.

In the GUI, enable Large Data Mode before running large files. The mode uses
file-backed Data_Load intake, turns off expensive optional refit diagnostics by
default, adds memory and dtype audit tables, trains on a configurable governed
sample, scores the full file in chunks, and lets you choose Parquet or sampled
tabular outputs.

Why downloaded HTML charts may fail from SageMaker
--------------------------------------------------

The interactive report uses dynamic Plotly charts. Static SVG fallback charts
are no longer embedded because they substantially increase report-build time.
On Linux, Kaleido v1 still requires Chrome if you enable optional per-figure
PNG exports.

If downloaded reports/interactive_report.html files show chart fallback
messages, open the report in Chrome or serve it from a local HTTP server. If
you specifically need optional PNG figure exports, rerun:

   bash scripts/bootstrap_sagemaker.sh

Then rerun the Quant Studio workflow with individual figure export enabled. To
skip the Chrome install attempt in locked-down environments, use:

   INSTALL_PLOTLY_CHROME=0 bash scripts/bootstrap_sagemaker.sh

Fallback: SageMaker JupyterLab browser proxy
--------------------------------------------

If you are not using local VS Code Remote and are instead running directly from
a SageMaker JupyterLab browser tab, start Streamlit with:

   BASE_URL_PATH=/jupyterlab/default/proxy/absolute/8501 bash scripts/run_sagemaker_streamlit.sh

Then open this URL pattern in the browser, using the same host and region from
your current JupyterLab tab:

   https://<your-space>.studio.<region>.sagemaker.aws/jupyterlab/default/proxy/absolute/8501/

Fallback: SageMaker Notebook Instance proxy
-------------------------------------------

If you are running from a SageMaker Notebook Instance instead of Studio
JupyterLab, use this command:

   BASE_URL_PATH=/proxy/absolute/8501 bash scripts/run_sagemaker_streamlit.sh

Then open:

   https://<notebook-name>.notebook.<region>.sagemaker.aws/proxy/absolute/8501/

If direct pip install does not work
-----------------------------------

Your SageMaker domain may not have outbound access to PyPI, or it may require a
private package source. In that case, use one of these:

1. A SageMaker lifecycle configuration that clones the repo and runs the
   bootstrap script automatically.
2. A custom SageMaker image with the dependencies preinstalled.
