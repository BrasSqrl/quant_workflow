Quant Studio walkthrough

Build a simple PD logistic regression model

This offline guide walks a new user through a full Quant Studio run: data setup, schema roles, binary PD logistic model configuration, readiness checks, results review, exports, and model artifact handoff.

Target outcome: a completed `fit_new_model` run using binary `logistic_regression`, with model diagnostics, an interactive report, a decision summary, and `model/quant_model.joblib` saved in the run folder.

Before You Start

Use the GUI

Launch Streamlit with the normal project scripts or command-line entry point, then open the local URL shown by Streamlit.

streamlit run app/streamlit_app.py

Use beginner defaults

Start with `Guided` workspace mode. Switch to `Advanced` only if you need extra controls.

Know the target

For PD modeling, use a binary target source such as `default_status`, where 1 or default-like values mean the event occurred.

1

Launch Quant Studio

Open the app and confirm the five workflow steps appear across the top. Quant Studio opens to Step 1 by default.

  • Local Windows users can use the project launch script.
  • macOS users should follow `docs/MACOS_SETUP.md`.
  • SageMaker users should use VS Code Remote port forwarding from `docs/SAGEMAKER_SETUP.md`.
If the app opens but looks stale, hard refresh the browser. Streamlit caches some frontend state between reruns.
2

Select The Dataset

In Step 1, choose the bundled sample data for the first run, or select a CSV/Excel/Parquet file from `Data_Load`, local path, browser upload, or S3 when Large Data Mode applies.

Bundled sample

Best for learning the workflow. The sample is synthetic panel loan data with financial statement fields, borrower behavior fields, dates, IDs, and a binary default target.

User data

For real PD data, include one target source, candidate features known before the prediction date, and date/entity fields for panel or out-of-time workflows.

3

Prepare Dataset & Schema

Use the Step 1 section controls in this order: `Dataset Preview`, `Data Review`, `Column Designer`, `Feature Dictionary`, `Transformation Studio`, and `Template Workbook`.

SectionWhat To Confirm
Data ReviewReview data contract, leakage-name flags, duplicate indicators, target distribution, and schema fingerprint.
Column DesignerMark `default_status` as `target_source`, the observation date as `date`, the loan/customer ID as `identifier`, and modeling columns as `feature`.
Feature DictionaryAdd business definitions, expected sign, unit, source system, and rationale for important features.
Transformation StudioOptional for a simple logistic model. Use recommendations or recipes only when a transform is defensible. Scorecard/WOE workflows can add bins or WOE here.
Template WorkbookDownload/upload the review workbook only when offline governance edits are needed.
For this simple walkthrough, do not over-transform the data. A plain logistic regression is easiest to validate when the feature set and preprocessing choices are simple.
4

Configure The Logistic Regression Model

Click Step 2 `Model Configuration`. Use the grouped controls to set the model run.

Core setup

  • `Execution mode`: `fit_new_model`
  • `Target mode`: `binary`
  • `Model family`: binary logistic / PD
  • `Model type`: `logistic_regression`
  • `Data structure`: `panel` if each borrower has repeated observations; otherwise `cross_sectional`

Safe beginner defaults

  • Use automatic or date-aware split settings unless your policy requires explicit windows.
  • Keep default diagnostics on for the first run.
  • Leave advanced visual analytics off unless exploratory visuals are needed.
  • Use default export settings until the first artifact package is reviewed.
If the bundled sample defaults to scorecard logistic regression, switch the model type to plain `logistic_regression` for this walkthrough. Scorecard logistic regression is useful later when you want binning, WOE, score points, and reason codes.
5

Run Readiness Checks

Click Step 3 `Readiness Check & Run`. Resolve blockers before running. Warnings can be acceptable when they are understood and documented.

ReadinessTarget, split, model compatibility
ResourcesMemory, disk, Large Data Mode
Checkpoint FlowStage-by-stage execution status
Click `Run Quant Model Workflow` when readiness has no blockers. The full workflow still runs through discrete checkpoint stages so memory can be released between major phases.
6

Review Results And Diagnostics

After completion, Step 4 `Results & Artifacts` shows grouped outputs. Start with model performance, then calibration and stability.

ROC / AUC

AUC measures rank ordering. Higher is generally better, but confirm stability and calibration.

KS

KS shows separation between event and non-event score distributions.

Calibration

Calibration checks whether predicted PD levels align with observed default rates.

Also review lift/gain, threshold diagnostics, feature importance, statistical tests, data quality, stability, and any skipped diagnostics. Skips are not always failures; they often mean the selected data or model type does not support that specific diagnostic.

7

Use The Decision Summary And Downloads

Click Step 5 `Decision Summary` for the decision-ready synthesis. This page is the fastest way to see whether the run is ready for review, needs remediation, or needs additional explanation.

Primary review items

  • Recommendation and attention items
  • Primary metric scorecard
  • Top feature drivers
  • Validation checklist
  • Evidence traceability map

Step 5 export buttons

  • `Download decision summary` for a portable Markdown scorecard.
  • `Download Individual Images` for standalone chart PNG/HTML files on demand.
  • `Download LLM Package` for curated evidence used in methodology-document drafting.
  • `Download OM Package` for the separate ongoing-monitoring handoff bundle.
8

Find The Model Object And Rerun Stack

The run folder is written under the configured artifact root, usually `artifacts/`, with a timestamped name.

artifacts/
  run_YYYY-MM-DD_HH-MM-SS_UTC/
    START_HERE.md
    model/quant_model.joblib
    config/run_config.json
    code/generated_run.py
    code/HOW_TO_RERUN.md
    reports/interactive_report.html
    reports/decision_summary.md
    artifact_manifest.json

Use `model/quant_model.joblib` as the fitted model object for score-existing-model workflows or standalone Python scoring. Use `config/run_config.json` and `code/generated_run.py` when the full workflow/config needs to be recreated outside the GUI.

Acceptance Checklist

QuestionExpected Answer
Did the run complete?Step 3 checkpoint flow shows completed stages and Step 4/5 display current outputs.
Was the model fit as logistic regression?Run configuration and summary identify `logistic_regression` with binary target mode.
Are performance diagnostics available?ROC/AUC, KS, calibration, lift/gain, and threshold diagnostics are present when labels are available.
Can the run be audited?`START_HERE.md`, `artifact_manifest.json`, `config/run_config.json`, `metadata/reproducibility_manifest.json`, and reports are exported.
Can the model be reused?`model/quant_model.joblib` is present and can be loaded by the score-existing-model workflow.
